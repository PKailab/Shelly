# Background Agents Design Spec

**Date:** 2026-04-05 (rev.2 — post-review fixes)
**Goal:** Autonomous background task execution framework for Shelly. Agents run in Termux, collect information, execute workflows, and deliver results — all without user interaction after initial setup.

**Primary Use Cases:**
1. Information curation — collect AI/STEAM/education papers and articles from the web, summarize with evidence, save to Obsidian vault on a schedule
2. OSS monitoring — watch specified repos for PRs/issues, analyze changes, produce summaries with fix proposals

**Scope:** Termux-local execution only. No cloud infrastructure. No paid API calls except Perplexity (academic search).

---

## Architecture Overview

```
User (Chat UI with @agent commands)
  → AgentManager (orchestrator)
    → AgentScheduler (AlarmManager primary, crond fallback)
    → AgentExecutor (runs tasks in tmux sessions)
      → ToolRouter (selects CLI/LLM per task)
        → Claude Code CLI (claude --print)
        → Gemini CLI
        → Local LLM (llama-server)
        → Perplexity API (curl)
      → OutputWriter (saves results to specified directory)
    → NotificationService (expo-notifications)
```

### Key Principles
- Agents run in **tmux sessions** (survives app backgrounding, proven by Immortal Sessions)
- Each agent task is a **self-contained script** generated by the LLM and executed in a tmux pane
- No paid API except Perplexity (academic/research search)
- User always approves: tool selection proposal, schedule, output destination
- **v1: Chat UI only** (`@agent` commands). Terminal command interception (`shelly agent *`) deferred to v2 (no command interception layer exists yet)

## 1. Agent Definition

An agent is a reusable task template with a schedule and configuration.

```typescript
interface Agent {
  id: string;
  name: string;                    // "AI Paper Curator"
  description: string;             // "Collects AI/STEAM papers weekly"
  prompt: string;                  // The task instruction
  schedule: CronExpression | null; // null = manual trigger only
  tool: ToolChoice;                // Which LLM/CLI to use
  outputPath: string;              // e.g. "~/Obsidian/AI-Papers/"
  outputTemplate: string | null;   // Markdown template for results
  enabled: boolean;
  lastRun: number | null;          // timestamp
  lastResult: 'success' | 'error' | null;
  createdAt: number;
}

type ToolChoice = 
  | { type: 'cli'; cli: 'claude' | 'gemini' | 'codex' }
  | { type: 'local' }
  | { type: 'perplexity' }
  | { type: 'auto' };  // Shelly suggests based on task content

type CronExpression = string;  // e.g. "0 9 * * 1,3,5" (Mon/Wed/Fri 9am)
```

### Agent Storage
- Agents stored in `~/.shelly/agents/` as JSON files
- One file per agent: `~/.shelly/agents/{id}.json`
- Agent list loaded at app startup into Zustand store

### Run History
- Each run logged to `~/.shelly/agents/logs/{agent-id}/{timestamp}.json`
- Contains: result status, output preview (first 500 chars), duration, tool used, error message if failed
- Last 30 runs retained per agent, older logs auto-pruned

## 2. Tool Router

Selects the appropriate CLI/LLM for each task. When `tool.type === 'auto'`, analyzes the prompt and suggests.

### Selection Logic (auto mode)
```
Priority-ordered keyword matching:
  1. "paper", "research", "study", "evidence", "journal", "academic", "論文"
     → Suggest: Perplexity API
  2. "PR", "issue", "commit", "repo", "code review", "GitHub"
     → Suggest: Claude Code CLI (or Gemini CLI if claude unavailable)
  3. "summarize", "format", "translate", "rewrite", "要約", "整形"
     → Suggest: Local LLM (if running) or Gemini CLI
  4. Default: Gemini CLI (free, general-purpose)

If multiple categories match, highest priority wins (1 > 2 > 3 > 4).
```

### Tool Availability Check
Before suggesting, verify the tool is available:
- CLI tools: `which claude`, `which gemini`, `which codex` via bridge
- Local LLM: check llama-server at configured endpoint
- Perplexity: check API key exists in `~/.shelly/agents/.env`

### User Approval Flow
```
Shelly: "This task looks like academic research. I recommend Perplexity API.
         Alternatively: Gemini CLI (free, less search depth).
         Which would you like to use?"
User: "Perplexity"
→ Tool locked to Perplexity for this agent
```

## 3. Agent Executor

Runs agent tasks in isolated tmux sessions.

### Execution Flow
```
1. AgentScheduler triggers (AlarmManager or manual)
2. Check concurrency lock (~/.shelly/agents/locks/{agent-id}.pid)
   - If locked and process alive → skip this run, log "skipped: previous run still active"
   - If locked but process dead → remove stale lock, proceed
3. Acquire lock (write PID to lockfile)
4. AgentExecutor creates a new tmux session: "shelly-agent-{id}"
5. Generates execution script (run-agent.sh) based on agent config:
   - Sources environment: PATH, HOME, API keys from ~/.shelly/agents/.env
   - For CLI tools: constructs a shell command
   - For Perplexity API: generates a curl command
   - For Local LLM: sends prompt to llama-server endpoint
6. Captures output (stdout redirect to $HOME/.shelly/tmp/agent-result-{id}.md)
7. Formats output using template (if specified)
8. Writes result to outputPath
9. Logs run result to ~/.shelly/agents/logs/
10. Sends Android notification via expo-notifications
11. Updates agent.lastRun and agent.lastResult
12. Releases lock, destroys tmux session
```

### CLI Command Construction
```bash
#!/bin/bash
# run-agent.sh — generated per agent, called by scheduler
# Source environment (PATH, HOME, API keys)
source "$HOME/.shelly/agents/.env"
export PATH="$PREFIX/bin:$PATH"
export HOME="/data/data/com.termux/files/home"

RESULT_FILE="$HOME/.shelly/tmp/agent-result-$AGENT_ID.md"

# Claude Code
claude --print "$PROMPT" > "$RESULT_FILE" 2>&1

# OR Gemini CLI
gemini "$PROMPT" > "$RESULT_FILE" 2>&1

# OR Perplexity API
curl -s https://api.perplexity.ai/chat/completions \
  -H "Authorization: Bearer $PERPLEXITY_API_KEY" \
  -H "Content-Type: application/json" \
  -d "{\"model\":\"sonar\",\"messages\":[{\"role\":\"user\",\"content\":\"$PROMPT\"}]}" \
  | jq -r '.choices[0].message.content' > "$RESULT_FILE"
```

### Error Handling
- CLI timeout: 10 minutes default, configurable per agent
- API failure: retry once after 30s, then mark as error
- tmux session crash: detect via `tmux has-session`, mark as error
- All errors → Android notification with error summary
- Lock cleanup: always release lock in trap/finally, even on error

## 4. Agent Scheduler

Manages scheduled execution for recurring agents.

### Primary: AlarmManager (via Expo Native Module)

AlarmManager survives Android Doze mode with exact alarms and is the reliable choice on Android 12+.

```kotlin
// In TerminalEmulatorModule.kt — add new AsyncFunctions
AsyncFunction("scheduleAgent") { agentId: String, intervalMs: Long, triggerAtMs: Long ->
    val context = appContext.reactContext ?: return@AsyncFunction null
    val am = context.getSystemService(Context.ALARM_SERVICE) as AlarmManager
    val intent = Intent(context, AgentAlarmReceiver::class.java).apply {
        putExtra("agent_id", agentId)
    }
    val pi = PendingIntent.getBroadcast(
        context, agentId.hashCode(), intent,
        PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE
    )
    am.setRepeating(AlarmManager.RTC_WAKEUP, triggerAtMs, intervalMs, pi)
    null
}

AsyncFunction("cancelAgent") { agentId: String ->
    // Cancel the PendingIntent
}
```

```kotlin
// AgentAlarmReceiver.kt — BroadcastReceiver
class AgentAlarmReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val agentId = intent.getStringExtra("agent_id") ?: return
        // Start Termux RunCommandService to execute run-agent.sh
        val runIntent = Intent("com.termux.RUN_COMMAND").apply {
            setClassName("com.termux", "com.termux.app.RunCommandService")
            putExtra("com.termux.RUN_COMMAND_PATH", "$HOME/.shelly/agents/run-agent.sh")
            putExtra("com.termux.RUN_COMMAND_ARGUMENTS", arrayOf(agentId))
            putExtra("com.termux.RUN_COMMAND_BACKGROUND", true)
        }
        context.startService(runIntent)
    }
}
```

### Fallback: Termux crontab
For users who prefer cron or when AlarmManager is unreliable:
```bash
pkg install cronie  # One-time setup
# Entry added by Shelly:
# 0 9 * * 1,3,5 ~/.shelly/agents/run-agent.sh agent-id-123
```

Note: crond must be started after each Termux restart. Termux:Boot can automate this via `~/.termux/boot/start-crond.sh`.

### Natural Language → Cron/Interval Conversion
When user says "毎週月水金の朝9時":
1. Shelly sends to available LLM: "Convert to cron expression: every Monday, Wednesday, Friday at 9am"
2. LLM returns: `0 9 * * 1,3,5`
3. Shelly shows: "スケジュール: 月・水・金 09:00 (0 9 * * 1,3,5). 正しいですか？"
4. User confirms → register with AlarmManager (and optionally crontab)

### Schedule Management
```typescript
interface AgentScheduler {
  install(agent: Agent): Promise<void>;    // Register AlarmManager + optional crontab
  uninstall(agentId: string): Promise<void>; // Cancel alarm + remove crontab entry
  listScheduled(): Promise<Agent[]>;
  runNow(agentId: string): Promise<void>;   // Manual trigger
}
```

## 5. API Key Management for Headless Execution

### Problem
SecureStore (expo-secure-store) is only accessible from within the React Native app process. Cron/AlarmManager-triggered shell scripts cannot access SecureStore.

### Solution
API keys needed by agents are exported to a chmod 600 file:

```
~/.shelly/agents/.env
```

```bash
# Auto-generated by Shelly. Do not edit manually.
# Permissions: -rw------- (owner read/write only)
PERPLEXITY_API_KEY=pplx-xxxxxxxxxxxx
# Future keys added here as needed
```

### Sync Mechanism
- When user sets/updates Perplexity API key in Settings, Shelly writes to both SecureStore (app use) and `.env` (agent use)
- `.env` file created with `chmod 600` (Termux UID only)
- On app startup, verify `.env` is in sync with SecureStore; update if stale
- `.env` is gitignored (add to project .gitignore)

### Security Assessment
- `.env` file is readable only by Termux UID (same sandbox as all terminal commands)
- Same security level as `~/.bashrc` containing `export API_KEY=...` (common developer practice)
- Not as secure as Android Keystore/SecureStore, but necessary for headless execution
- Acceptable tradeoff documented in spec

## 7. Output Writer

Saves agent results to user-specified locations.

### Output Path
- Absolute path: `~/Obsidian/AI-Papers/`
- Relative to project: `./reports/`
- Auto-create directories if needed (`mkdir -p`)

### Filename Generation
```
{outputPath}/{date}-{agent-name-slug}.md

Example:
~/Obsidian/AI-Papers/2026-04-05-ai-paper-curator.md
```

### Output Template
User can specify a Markdown template. Rendering uses simple `{{variable}}` string replacement (no template engine dependency).

Available variables:
```
{{date}}        — YYYY-MM-DD
{{timestamp}}   — ISO 8601
{{agent_name}}  — Agent display name
{{tool_used}}   — "Perplexity API" / "Gemini CLI" / etc.
{{content}}     — Raw LLM/CLI output
```

Default template (when none specified):
```markdown
---
date: {{date}}
agent: {{agent_name}}
tool: {{tool_used}}
---

# {{agent_name}} — {{date}}

{{content}}

---
*Generated by Shelly Background Agent at {{timestamp}}*
```

## 8. Notification Service

Uses `expo-notifications` (already installed as dependency).

### Notification Types
- **Task complete (success):** "✅ AI教育論文キュレーター completed. 5 papers saved."
- **Task failed:** "❌ AI教育論文キュレーター failed: Perplexity API timeout"
- **Task skipped:** "⏭️ AI教育論文キュレーター skipped: previous run still active"

### Implementation
```typescript
import * as Notifications from 'expo-notifications';

async function notifyAgentResult(agent: Agent, result: 'success' | 'error' | 'skipped', summary: string) {
  await Notifications.scheduleNotificationAsync({
    content: {
      title: `${result === 'success' ? '✅' : result === 'error' ? '❌' : '⏭️'} ${agent.name}`,
      body: summary,
      data: { agentId: agent.id },
    },
    trigger: null, // immediate
  });
}
```

## 10. Concurrency Control

### Per-Agent Lock
```
~/.shelly/agents/locks/{agent-id}.pid
```

- Lock acquired: write PID to file before execution
- Lock released: delete file after execution (in trap/finally)
- Stale lock detection: read PID, check `kill -0 $PID`. If process dead, remove lock.
- Concurrent trigger while locked: skip run, log as "skipped", notify user

### Global Limit
- Maximum 2 agents running concurrently (prevent Termux overload)
- Queue excess triggers, execute when a slot frees up
- Configurable in settings (1-4)

## 11. Lifecycle Management

### App Startup
- Load agent definitions from `~/.shelly/agents/*.json`
- Verify AlarmManager registrations are in sync (re-register if missing)
- Clean up orphaned tmux sessions (`tmux ls | grep shelly-agent- | ...`)
- Prune old logs (keep last 30 per agent)

### Agent Disable/Delete
- Cancel AlarmManager alarm
- Remove crontab entry (if exists)
- Kill tmux session (if running)
- Delete lockfile
- Optionally delete logs and output files (ask user)

### App Uninstall / Data Clear
- AlarmManager alarms are automatically cancelled by Android on uninstall
- crontab entries persist (documented in Limitations)
- tmux sessions persist until Termux restarts (documented in Limitations)

### Schema Migration
- Agent JSON files include a `version` field (v1 for initial release)
- On load, check version. If outdated, run migration function and rewrite file.

## 13. Security Considerations

- Agent prompts are user-defined — no automatic execution of untrusted content
- Perplexity API key stored in SecureStore (app) + chmod 600 file (headless) — see Section 5
- File writes restricted to user-specified outputPath
- tmux sessions run under Termux UID (same sandbox as manual terminal use)
- Command safety check (existing `command-safety.ts`) applied to generated shell commands
- Agents cannot modify their own definition files
- `.env` file is gitignored

## 14. Limitations (v1)

- **No cloud execution** — agents only run when device has Termux available
- **No inter-agent communication** — agents are independent, cannot chain outputs
- **No web scraping** — relies on CLI/API search capabilities, not direct page fetching
- **Perplexity is the only paid API** — other search APIs not supported in v1
- **No GUI for template editing** — templates are edited as raw Markdown text
- **Terminal commands deferred** — `shelly agent *` commands require a command interception layer (v2). v1 uses chat `@agent` commands only
- **crontab orphans on uninstall** — crontab entries are not auto-cleaned if Shelly is uninstalled without disabling agents first
- **AlarmManager precision** — Android may delay exact alarms by up to 10 minutes in Doze mode. Use `setExactAndAllowWhileIdle()` for better precision (requires SCHEDULE_EXACT_ALARM permission on Android 12+)
