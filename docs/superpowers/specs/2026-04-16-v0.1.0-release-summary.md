## What changed, section by section

### 1. Paste pipeline (bug #91 + #94)

**Problem.** Shelly had five independent paste entry points (A–E in the
audit doc), each with its own CR/LF handling and bracketed-paste wrapping.
Four separate paste bugs (#27, #58, #81, #91) all came from the same root
cause: no single choke point meant every fix touched one path and left the
others drifting.

**Fix.** Session C produced a full audit of the five paths
(`docs/superpowers/specs/2026-04-16-paste-pipeline-audit.md`) and Session A
implemented the recommended design:

1. `TerminalEmulator.paste()` now **always** wraps the payload in
   `\e[200~..\e[201~` regardless of the DECSET 2004 flag. CRLF is
   normalized to LF (not CR) so readline sees the payload as a single
   paste event.
2. A new `TerminalView.pasteViaEmulator(String)` helper centralizes the
   shadow-buffer bookkeeping, `mLastImeCommitAt` stamping, and final
   flush. IME `commitText` multi-line paths and middle-click mouse paste
   both go through it.
3. `BaseInputConnection.commitText` detects paste-like payloads (2+ chars
   and either newline-containing or `length >= 16`) and routes them to
   `pasteViaEmulator` instead of the per-char `sendTextToTerminal` loop.
4. `HomeInitializer.kt` now writes `bind 'set enable-bracketed-paste on'`
   into the generated `.bashrc`, so readline correctly interprets the
   bracketed-paste sequence on a fresh session. `BASHRC_VERSION` bumped
   18 → 19 → 20 (the middle bump is Session B's #93/#95 work).

**Observability.** A new `ShellyPaste` logcat tag traces raw / sanitized
byte length, newline count, and the first 32 characters (with `↵⏎` escapes
for readability) at every paste site.

### 2. `execSubprocess` read loop (bug #70)

**Problem.** `shelly-exec.c` used `select()` + non-blocking reads and
treated `n <= 0` as EOF. This collapsed EAGAIN (spurious wake, transient
empty pipe) with genuine end-of-stream, so the parent occasionally
returned `exit=0 stdout=""` when the child was still mid-flight. The
symptom was Sidebar / FileTree / GitStatus badges reading empty, which
led to the "all JNI read-via-shell paths are broken" working theory and
the Wave F `readDir` workaround.

**Fix.** Distinguish `n == 0` (real EOF) from `n < 0 && errno in
{EAGAIN, EWOULDBLOCK, EINTR}` (retry on next select tick). Any other
errno is logged and treated as a fatal read error. The Wave F
`readDir` JNI bypass is still in place — it's strictly safer than
shell-reading — but shell-based commands are now reliable again,
which unblocks git status polling, auto-savepoint, and other
`execCommand()` callers.

**Observability.** EAGAIN counts are tracked per-stream and included in
the final exit log line (`ShellyExec` tag), so a regression is detectable
from a single logcat entry.

### 3. `/sdcard` access (bug #92)

**Problem.** Android 11+ Scoped Storage blocks raw `open(2)` on `/sdcard`
even with the legacy `READ/WRITE_EXTERNAL_STORAGE` perms. The "adb push a
script to `/sdcard/Download`, source it from the shell" workflow was
completely broken; the codex manual patch session on 2026-04-16 ran
straight into this wall.

**Fix.**

- `app.config.ts` and `android/app/src/main/AndroidManifest.xml` both
  declare `android.permission.MANAGE_EXTERNAL_STORAGE`. Shelly is
  distributed via GitHub Releases and F-Droid, not Play Store, so the
  all-files-access audit does not apply.
- `TerminalEmulatorModule.kt` exposes `hasAllFilesAccess()` (wraps
  `Environment.isExternalStorageManager()`) and `requestAllFilesAccess()`
  (fires `Settings.ACTION_MANAGE_APP_ALL_FILES_ACCESS_PERMISSION`
  Intent, with a generic-settings fallback).
- `lib/first-launch-setup.ts` calls `ensureAllFilesAccess()` on every
  launch. The call is idempotent — if the permission is already
  granted, it logs and returns; otherwise it fires the Intent to open
  system Settings.

**Observability.** `TerminalEmulator` tag logs the permission state
(`sdk=N granted=bool`) and every Intent fire / fallback, so a user's
"nothing happened" report can be diagnosed immediately.

### 4. Codex CLI and bash wrapper (bug #76 / #93 / #95)

These were handled by Session B in a single commit (8f44e01c). The fixes
layered on top of the Alpine minirootfs + proot work from Wave L
(856025ff).

- **bug #93** — `$HOME/bin/bash` is now a shim that calls
  `/system/bin/linker64 libbash.so "$@"`. Scripts that say `bash
  script.sh` or use a `#!/usr/bin/env bash` shebang now work.
- **bug #95** — the codex.js sed patch (which rewrites `spawn(binaryPath,
  ...)` to `spawn("proot", [binaryPath.replace(HOME, "/root"), ...])`)
  now runs synchronously after `npm install` completes and logs every
  step to `~/.shelly-cli/install.log`. A final `grep -q 'shelly-proot'`
  verification confirms the patch applied; any failure mode leaves a
  diagnostic line in the log instead of silently succeeding.
- **bug #76** — the combination of Wave L's rootfs + proot + Session B's
  sed timing fix should make `codex --version` work on first install.
  This is the only bug in the release that requires device verification
  to mark fully closed.

### 5. Sidebar repo probing (bug #73)

**Problem.** The Add Repository modal accepted any string, leading to
ghost entries that rendered empty file trees and zero git counts.
Typos and Termux-era paths were silently persisted.

**Fix.** `tryAddRepo` probes the parent directory via the JNI
`readDirEntries` path and checks whether the basename is present as a
directory or symlink. Nonexistent paths trigger an `Alert`; unreadable
parents fall through permissively (logs `"parent unreadable, accepting
add"` for diagnostics).

## Observability catalog

A new debug-log tag table lives in `CLAUDE.md`. For quick reference:

| Tag | What it traces |
|-----|----------------|
| `ShellyPaste` | All paste paths (commitText, middle-click, CommandKeyBar) |
| `ShellyIME` | Full IME event flow (commitText, setComposingText, delete storms) |
| `ShellyExec` | Child PID, exit code, stdout/stderr size, EAGAIN count |
| `ShellyPTY` | forkpty + termios initialization |
| `HomeInitializer` | `.bashrc` regeneration trigger decisions |
| `TerminalEmulator` | `hasAllFilesAccess` / `requestAllFilesAccess` state |
| `Sidebar` | `tryAddRepo` probe results |

Typical verification command:

```bash
adb logcat -s ShellyPaste:D ShellyIME:D ShellyExec:D HomeInitializer:D TerminalEmulator:D Sidebar:D
```

## Verification matrix — what to test on the release APK

| Scenario | Expected result |
|----------|-----------------|
| Paste a long single-line `sed` command into the terminal | One command arrives, one Enter executes it. logcat: `ShellyPaste pasteViaEmulator len=N nl=0` |
| Paste a multi-line script block | No per-line execution until the final `\n`. logcat: `ShellyPaste paste(..., nl=N, preview=...)` |
| `adb push /sdcard/Download/foo.sh` then `source /sdcard/Download/foo.sh` in shell | First launch asks for all-files-access; after grant, `source` reads successfully. |
| `bash script.sh` in shell | Runs via `$HOME/bin/bash` wrapper; `which bash` returns the wrapper path. |
| `codex --version` in shell | Runs via Alpine rootfs + proot; returns version string. Confirm by tailing `~/.shelly-cli/install.log` for the `[patch] codex.js patch verified OK` line. |
| `git status` badge in Sidebar | Shows correct dirty count (not `0` silently). logcat: `ShellyExec ... eagain=N` confirms non-blocking reads are retried when needed. |
| Add a nonexistent repo path in Sidebar | `Alert("Directory not found")`, entry not added. logcat: `Sidebar tryAddRepo probe ... exists=false` |
| YouTube fullscreen inside a Browser pane | Pane maximizes within its container; other panes / sidebar stay visible. |
| Terminal right edge padding | Text never touches the right border; a 6px gutter is always visible. |
